# PostgreSQL Schema Fundamentals

Core mechanics for creating and editing **tables**, **columns**, and **relationships** using `psql`. Examples assume you're connected like:

```
psql -U DaltonManning -h localhost -d Prod
```

---

## 1. The mental model

A **database** holds **tables**. A table is a grid: **columns** define the shape (name + data type + rules), **rows** are the data. **Relationships** are just columns in one table that point at the primary key of another table, enforced by a **foreign key**.

Three verbs do almost everything:

| Verb | Purpose |
|------|---------|
| `CREATE TABLE` | make a new structure |
| `ALTER TABLE`  | change an existing structure |
| `DROP TABLE`   | delete a structure entirely |

---

## 2. Creating a table

```sql
CREATE TABLE category (
    id          bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name        text NOT NULL UNIQUE,
    created_at  timestamptz NOT NULL DEFAULT now()
);
```

Each column line is `name  type  [constraints]`. Read it left to right:
`name` is the column, `text` is the data type, `NOT NULL UNIQUE` are the rules.

**Safe variants:**

```sql
CREATE TABLE IF NOT EXISTS category ( ... );   -- skip if it already exists
DROP TABLE IF EXISTS category;                 -- no error if it's missing
```

---

## 3. Common column data types

| Type | Use for | Example |
|------|---------|---------|
| `text` | strings of any length | `'Pressure'` |
| `varchar(n)` | length-capped strings | `varchar(50)` |
| `integer` / `int` | whole numbers | `42` |
| `bigint` | large whole numbers (IDs) | `9000000000` |
| `numeric(p,s)` | exact decimals (money, measurements) | `numeric(10,2)` |
| `boolean` | true/false | `true` |
| `timestamptz` | date+time with timezone | `now()` |
| `date` | calendar date only | `'2026-06-27'` |
| `jsonb` | structured/nested data | `'{"k":"v"}'` |
| `uuid` | globally unique IDs | `gen_random_uuid()` |

Prefer `text` over `varchar(n)` unless you genuinely need the length cap — in Postgres they perform identically and `text` saves you future `ALTER`s.

---

## 4. Column constraints (the rules)

| Constraint | Meaning |
|------------|---------|
| `NOT NULL` | value is required |
| `UNIQUE` | no duplicates allowed in this column |
| `PRIMARY KEY` | unique + not null; the row's identity |
| `DEFAULT <expr>` | auto-fill when no value given |
| `CHECK (<condition>)` | value must satisfy a rule |
| `GENERATED BY DEFAULT AS IDENTITY` | auto-incrementing number |
| `REFERENCES other(col)` | foreign key → see §6 |

Example combining several:

```sql
CREATE TABLE translation_table (
    id           bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    property     text NOT NULL,
    translation  text,
    category     text,
    priority     int  NOT NULL DEFAULT 0 CHECK (priority >= 0),
    loaded_at    timestamptz NOT NULL DEFAULT now()
);
```

## 5. Editing an existing table — `ALTER TABLE`

You rarely drop and rebuild in production. You alter in place. All of these are non-destructive to other columns.

**Add a column** (existing rows get the default or NULL):

```sql
ALTER TABLE translation_table ADD COLUMN notes text;
ALTER TABLE translation_table ADD COLUMN active boolean NOT NULL DEFAULT true;
```

**Rename a column:**

```sql
ALTER TABLE translation_table RENAME COLUMN notes TO comment;
```

**Change a column's type** (USING tells Postgres how to convert existing data):

```sql
ALTER TABLE translation_table ALTER COLUMN priority TYPE bigint;
ALTER TABLE translation_table ALTER COLUMN priority TYPE text USING priority::text;
```

**Add or drop NOT NULL:**

```sql
ALTER TABLE translation_table ALTER COLUMN translation SET NOT NULL;
ALTER TABLE translation_table ALTER COLUMN translation DROP NOT NULL;
```

**Change or drop a default:**

```sql
ALTER TABLE translation_table ALTER COLUMN active SET DEFAULT false;
ALTER TABLE translation_table ALTER COLUMN active DROP DEFAULT;
```

**Drop a column** (destroys that column's data — irreversible):

```sql
ALTER TABLE translation_table DROP COLUMN comment;
```

**Rename the whole table:**

```sql
ALTER TABLE translation_table RENAME TO translations;
```

## 6. Relationships — foreign keys

A relationship connects a **child** row to a **parent** row. The child holds a column whose value must match a primary key in the parent. This is a **one-to-many**: one category, many translations.

### Define it at creation time

```sql
CREATE TABLE category (
    id    bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name  text NOT NULL UNIQUE
);

CREATE TABLE translation_table (
    id           bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    property     text NOT NULL,
    translation  text,
    category_id  bigint REFERENCES category(id)   -- the foreign key
);
```

Now Postgres refuses any `category_id` that doesn't exist in `category`. That guarantee is the whole point — no orphaned rows.

### Add a foreign key to a table that already exists

```sql
ALTER TABLE translation_table
    ADD CONSTRAINT fk_category
    FOREIGN KEY (category_id) REFERENCES category(id);
```

### Control what happens when the parent is deleted

```sql
category_id bigint REFERENCES category(id) ON DELETE CASCADE,   -- delete children too
category_id bigint REFERENCES category(id) ON DELETE SET NULL,  -- null out the link
category_id bigint REFERENCES category(id) ON DELETE RESTRICT,  -- block the delete (default)
```

### Many-to-many: use a join table

When rows on both sides can link to many of the other (e.g. a translation belongs to several tags, a tag covers several translations), make a third table holding one row per pairing:

```sql
CREATE TABLE tag (
    id   bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name text NOT NULL UNIQUE
);

CREATE TABLE translation_tag (
    translation_id bigint REFERENCES translation_table(id) ON DELETE CASCADE,
    tag_id         bigint REFERENCES tag(id) ON DELETE CASCADE,
    PRIMARY KEY (translation_id, tag_id)   -- composite key prevents duplicate pairs
);
```

### Drop a relationship

```sql
ALTER TABLE translation_table DROP CONSTRAINT fk_category;
```

## 7. Indexes (speed, not structure)

A foreign key or filtered column gets queried a lot — add an index so lookups stay fast as the table grows. `PRIMARY KEY` and `UNIQUE` create indexes automatically; everything else you add yourself.

```sql
CREATE INDEX idx_tt_category_id ON translation_table (category_id);
CREATE INDEX idx_tt_property    ON translation_table (property);
DROP INDEX idx_tt_property;
```

## 8. Inspecting what you built — `psql` meta-commands

| Command | Shows |
|---------|-------|
| `\dt` | all tables |
| `\d translation_table` | columns, types, constraints, indexes, foreign keys |
| `\di` | all indexes |
| `\d+ translation_table` | the above plus storage/size detail |
| `\x` | toggle expanded (vertical) row display — good for wide rows |
| `\q` | quit psql |

`\d <table>` is the one you'll use constantly: it lists every column with its type and nullability, then the keys and foreign-key references underneath. It's the fastest way to confirm an `ALTER` did what you expected.

## 9. Quick reference — the whole loop

```sql
-- create
CREATE TABLE thing (id bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY, name text NOT NULL);

-- inspect
\d thing

-- edit structure
ALTER TABLE thing ADD COLUMN qty int NOT NULL DEFAULT 0;
ALTER TABLE thing RENAME COLUMN name TO label;

-- relate
ALTER TABLE child ADD CONSTRAINT fk FOREIGN KEY (thing_id) REFERENCES thing(id);

-- remove
ALTER TABLE thing DROP COLUMN qty;
DROP TABLE IF EXISTS thing;
```

---

### Two habits worth keeping

1. **`ALTER`, don't drop/rebuild, once a table has real data.** Dropping destroys rows; altering preserves them.
2. **Wrap risky structural changes in a transaction** so you can back out if something errors midway:

   ```sql
   BEGIN;
     ALTER TABLE translation_table ADD COLUMN category_id bigint;
     ALTER TABLE translation_table ADD CONSTRAINT fk_category
         FOREIGN KEY (category_id) REFERENCES category(id);
   COMMIT;   -- or ROLLBACK; to undo everything since BEGIN
   ```
